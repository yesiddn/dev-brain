## Contexto

> ¿Cuándo usar este playbook?
> Este playbook es útil para desarrolladores que quieren iniciar un proyecto de dotnet de manera rápida y sencilla.

## Pasos

### 1. Crear proyectos

Una vez tengas el proyecto base creado, pasamos a crear varios proyectos de tipo biblioteca de clases para separar responsabilidades.
En este caso vamos a usar un estructura simple de cuatro proyectos:

- Api (webapi): Este seria el proyecto con el que se inicia y es el encargado de los endpoints y binding.
- Application (classLib): Se encarga de los casos de uso y lógica de negocio (DTOs y servicios).
- Domain (classLib): Se encarga de representar conceptos de negocio (entidades).
- Infrastructure (classLib): Se encarga de resolver detalles técnicos externos (base de datos).

Para crear los proyectos se usan estos comandos:

```bash
# crear proyecto tipo webapi
dotnet new webapi -n NombreProyecto -o CarpetaDestino

# crear proyecto tipo classLib
dotnet new classLib -n NombreProyecto -o CarpetaDestino
```

Una vez tengamos los proyectos listos, hay que agregarlos a la Solución (en mi caso mi solución se llama `ceres`).

```bash
dotnet sln ceres.sln add ceres.Api/<classLib>.Api.csproj
```

> 💡 Si estas usando un IDE como Visual Studio o Rider puedes crear los proyectos desde la UI y el editor se encaarga de agregarlos a la solución.

Ahora, para poder usar las clases de un proyecto en otro proyecto, hay que referenciar el primer proyecto. Por ejemplo, si `Api` va a usar clases que viven en `Application`, como DTOs, entonces `Api`` necesita una referencia de proyecto hacia `Application`. Para hacer esto, ejecuta el siguiente comando:

```bash
dotnet add ceres.Api/<classLib>.Api.csproj reference ceres.Application/<classLib>.Application.csproj
```

### 2. Conexión a la base de datos

Para este playbook usaré PostgreSQL como base de datos.

#### 2.1. Instalar paquetes en `Infrastructure`

Para la configuración primero hay que instalar los paquetes necesrios.

```bash
dotnet add Ceres.Infrastructure package Microsoft.EntityFrameworkCore
dotnet add Ceres.Infrastructure package Npgsql.EntityFrameworkCore.PostgreSQL
dotnet add Ceres.Infrastructure package Microsoft.EntityFrameworkCore.Design
```

> 💡 Si usas un IDE como Visual Studio o Rider puedes usar la UI de Nuget para buscar los paquetes e instalarlos con un click.

#### 2.2. Crear un `AppDbContext.cs`

En `Ceres.Infrastructure/Persistence/AppDbContext.cs`:

```csharp
using Microsoft.EntityFrameworkCore;

namespace Ceres.Infrastructure.Persistence;

public class AppDbContext : DbContext
{
    public AppDbContext(DbContextOptions<AppDbContext> options) : base(options)
    {
    }
}
```

#### 2.3. Agregar connection string

En `Ceres.Api/appsettings.Development.json`:

```json
{
  "ConnectionStrings": {
    "DefaultConnection": "Host=localhost;Port=5432;Database=ceresdb;Username=postgres;Password=postgres"
  }
}
```

Ese es el formato típico de connection string para PostgreSQL.

#### 2.4. Registra el `DbContext` en `Program.cs`

En `Ceres.Api/Program.cs`:

```csharp
using Ceres.Infrastructure.Persistence;
using Microsoft.EntityFrameworkCore;

var builder = WebApplication.CreateBuilder(args);

builder.Services.AddDbContext<AppDbContext>(options =>
    options.UseNpgsql(builder.Configuration.GetConnectionString("DefaultConnection")));

var app = builder.Build();

app.Run();
```

`UseNpgsql` es la configuración correcta para PostgreSQL en EF Core. Si no reconoce `UseNpgsql`, revisa el paquete y el `using Microsoft.EntityFrameworkCore;`.

> No olvides agregar la referencia de `Infrastructure` para poder usar el `AppDbContext`.
